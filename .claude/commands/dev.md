# WBS 기반 개발

WBS 문서에서 해당 태스크를 추출하여 개발을 진행합니다.

## 입력 처리

입력: $ARGUMENTS

**처리 순서**:
1. 입력이 WBS 번호 패턴(예: "2.1", "3.2.1")인지 확인
2. WBS 번호인 경우: `docs/PDF_VALIDATION_WBS.md`에서 해당 섹션 추출
3. 일반 텍스트인 경우: 기존 feature-dev 워크플로우 진행

---

## Phase 1: WBS 번호 파싱

WBS 번호가 감지되면 (`/^\d+\.\d+(\.\d+)?$/` 패턴):

1. `docs/PDF_VALIDATION_WBS.md` 파일 읽기
2. 해당 WBS 섹션(예: "### WBS 2.1:")부터 다음 "### WBS" 또는 "## WBS" 전까지 추출
3. 추출된 내용 요약

추출할 정보:
- 작업 설명 (| 작업 | ... |)
- 대상 파일 (| 파일 | ... |)
- 예상 소요 시간
- 서브태스크 목록 (`- [ ]` 항목)
- 상세 구현 코드 블록

---

## Phase 2: 태스크 확인

추출된 WBS 내용을 분석하여:
1. 작업 범위 요약
2. 수정할 파일 목록
3. 서브태스크를 TodoWrite로 등록

---

## Phase 3: 코드 구현

1. 대상 파일 읽기
2. WBS에 제공된 상세 구현 코드 적용
3. 기존 코드 스타일/패턴 준수
4. 서브태스크 순서대로 구현

---

## Phase 4: 테스트 작성

1. WBS에 명시된 테스트 요구사항 확인
2. 단위 테스트 작성/확장
3. 테스트 실행 및 통과 확인

---

## Phase 5: 완료 보고

1. 모든 서브태스크 완료 확인
2. 구현 결과 요약:
   - 수정된 파일
   - 추가된 기능
   - 테스트 결과
3. 다음 WBS 작업 제안 (의존성 기반)

---

## 사용 예시

- `/dev 2.1` → 가로형 페이지 감지 기능 개발
- `/dev 3.2` → GS inkcov 분석 기능 개발
- `/dev 사용자 인증 기능` → 일반 개발 워크플로우

---

## 일반 개발 (WBS 번호가 아닌 경우)

WBS 번호가 아닌 경우 feature-dev 워크플로우 진행:

1. **Discovery**: 요구사항 파악
2. **Codebase Exploration**: 관련 코드 탐색
3. **Clarifying Questions**: 불명확한 부분 질문
4. **Architecture Design**: 구현 방안 설계
5. **Implementation**: 코드 구현
6. **Quality Review**: 코드 리뷰
7. **Summary**: 완료 보고
